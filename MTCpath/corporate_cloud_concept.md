# Концепция: Corporate Cloud как функция MTC Cloud IaaS

## 1) Цель функции

`Corporate Cloud` — это дополнительная функция в экосистеме `MTC Cloud`, ориентированная **только на корпоративных пользователей**.

Сценарий использования:
- компания разворачивает сервер в `Proxmox`;
- на этом сервере запускается корпоративное облако;
- сотрудники компании подключаются к нему через мобильное приложение.

Функция не заменяет IaaS-платформу, а повышает её ценность: помимо VPS/VDS компания получает удобный слой совместной работы с файлами.

---

## 2) Ключевые роли

### 2.1 Обычный корпоративный пользователь (Employee)
- Создаёт профиль.
- Выполняет подключение к корпоративному облаку через **уникальную ссылку**.
- Вводит **ключ доступа**.
- Работает с файлами согласно выданным правам.

### 2.2 Администратор корпоративных файлов (Corporate Admin / Manager)
- Имеет все возможности Employee.
- Дополнительно получает доступ к разделу **«Панель администратора»** во вкладке **«Ещё»**.
- Управляет структурой корпоративных файлов и правами доступа.

---

## 3) Пользовательский путь (end-to-end)

## 3.1 Начальный экран (Welcome)
Назначение:
- объяснить, что это корпоративное облако в рамках MTC Cloud;
- дать быстрые действия: `Регистрация` / `Вход`.

Состояния:
- пользователь не авторизован → доступна регистрация/вход;
- пользователь авторизован, но не подключён к корпоративному облаку → показать путь к импорту по ссылке.

## 3.2 Регистрация / вход
Минимум данных профиля:
- username;
- имя/фамилия;
- email/телефон;
- пароль.

После успешной аутентификации:
- если корпоративная связь ещё не настроена, пользователь направляется на экран **импорта по ссылке**;
- если связь уже настроена, открывается основной экран с нижней навигацией.

## 3.3 Импорт по ссылке (уникальная ссылка + ключ доступа)
Экран подключает сотрудника к конкретному корпоративному облаку.

Поля:
- `Уникальная ссылка` (или авто-подстановка из deep-link);
- `Ключ доступа`.

Поведение:
- валидация формата ссылки и ключа;
- проверка доступа;
- привязка профиля пользователя к корпоративному облаку;
- переход в основной интерфейс при успехе.

Примечание для MVP хакатона:
- на текущем этапе нет готового backend-модуля для ссылок и MinIO;
- экран и UX фиксируются как концепция, а серверная проверка будет добавлена позже.

Ошибки:
- ссылка недействительна;
- ключ неверный;
- ключ просрочен;
- пользователь уже привязан к другой корпоративной организации (по политике безопасности).

## 4) Основная навигация (Bottom Nav)

## 4.1 Базовый набор вкладок
Для корпоративного пользователя:
1. `Файлы`
2. `+`
3. `Ещё`

## 4.2 Админ-раздел во вкладке «Ещё»
Если у пользователя есть право администратора корпоративных файлов:
- во вкладке `Ещё` отображается пункт `Панель администратора`.

Принцип:
- пункт отображается строго на основе роли/прав;
- без права пункт отсутствует полностью (не disabled, а скрыт).

---

## 5) Функции вкладок

## 5.1 Вкладка «Файлы»
Назначение: единая точка доступа к рабочему контенту.

Содержимое:
- список папок и файлов;
- обязательные зоны:
  - `Private` (личная область сотрудника);
  - корпоративные папки (shared/team/department и т.п.).

Базовые действия пользователя:
- открыть папку/файл;
- загрузить/скачать файл;
- поиск по названию;
- фильтрация по типам (документы, медиа, архивы — опционально для следующих итераций);
- просмотр недавних файлов.

Права доступа:
- чтение/запись/удаление зависят от роли пользователя и ACL папки.

## 5.2 Вкладка «+»
Назначение: быстрые операции добавления.

MVP-действия:
- загрузить файл;
- создать папку;
- импортировать медиа из устройства.

Поведение загрузки через `+`:
- `Windows (popup)`: пользователь нажимает `+` и перетягивает файл в выделенную drop-зону;
- `iOS/Android`: пользователь нажимает `+` и выбирает файл/медиа через системный picker.

Acceptance criteria (MVP):
1. **Успешная загрузка**: после drop/выбора файла появляется индикатор прогресса, а по завершении файл отображается в целевой папке.
2. **Ошибка формата**: если формат не поддерживается, показывается понятное сообщение об ошибке, загрузка не стартует.
3. **Прогресс загрузки**: пользователь видит статус (в процентах или этапах), пока файл загружается.
4. **Повтор попытки**: при сетевой/серверной ошибке доступна кнопка `Повторить` без повторного выбора файла в рамках текущей сессии.

Дальше (опционально):
- скан документа;
- массовая загрузка;
- шаблоны структуры папок.

## 5.3 Вкладка «Ещё»
Назначение: профиль и служебные функции.

Содержимое:
- профиль пользователя;
- сведения о подключённой организации;
- настройки (язык, тема, уведомления);
- безопасность (смена пароля, выход из аккаунта);
- `Панель администратора` (только для роли Manager/Corporate Admin);
- справка/поддержка.

## 5.4 Раздел «Панель администратора» (внутри «Ещё», только для админа)
Назначение: управление корпоративным пространством.

MVP-функции:
- менеджмент пользователей;
- назначение прав доступа для обычных пользователей;
- статистика использования;
- индикатор заполненности хранилища.

Следующие итерации:
- расширенный аудит действий;
- квоты по отделам/командам;
- шаблоны ролей и групповые политики.

---

## 6) Логика прав (RBAC/ACL)

Модель для хакатонного MVP:
- один пользователь состоит только в одной корпоративной организации;
- уровень организации: роль пользователя (`Employee`, `Corporate Admin`);
- уровень ресурса: права на папки назначаются менеджером;
- self-service запросов прав нет;
- права задаются на стороне базы/админ-логики.

Правило приоритета:
- отсутствие явного разрешения = запрет;
- админ корпоративных файлов не обязан быть платформенным супер-админом IaaS.

---

## 7) Интеграция с MTC Cloud IaaS

Функция позиционируется как value-add:
- компания создаёт инфраструктуру в IaaS (`Proxmox`);
- поверх инфраструктуры запускает корпоративное файловое пространство;
- сотрудники работают через удобный мобильный клиент.

Польза для платформы:
- повышение удержания корпоративных клиентов;
- дополнительный сценарий потребления ресурсов (хранение + трафик + управление доступом);
- расширение экосистемы без изменения базового IaaS-позиционирования.

---

## 7.1 Форматы клиентов (Mobile + Windows popup)

Поддерживаемые клиентские форматы в рамках концепции:
- `iOS/Android`: основное мобильное приложение (единый Flutter-клиент).
- `Windows`: mini-app только в формате `popup` (иконка в трее, по нажатию открывается компактное окно в стиле обычного приложения).

Важно:
- это не системный Windows Widget из панели Widgets, а отдельное desktop-приложение с похожим UX;
- для хакатонного MVP используется только `popup`-режим;
- отдельная адаптация под широкий desktop-layout (16:9 и т.п.) не требуется.

---

## 8) MVP-рамки (без реализации кода на текущем этапе)

В первую версию концептуально входят:
- регистрация/вход корпоративного пользователя;
- импорт в организацию по уникальной ссылке и ключу доступа;
- нижняя навигация `Файлы / + / Ещё`;
- пункт `Панель администратора` внутри `Ещё` (только для админа);
- разделение `Private` и корпоративных папок;
- базовые операции с файлами.

Не входят в MVP (можно позже):
- расширенная автоматизация политик;
- self-service workflow согласований прав;
- полноценный биллинг внутри мобильного модуля;
- offline-доступ к `Private` файлам.

---

## 9) Зафиксированные решения по ТЗ

1. На старте пользователь может состоять только в одной корпоративной организации.
2. При проблеме с ключом: повторный импорт по ссылке; если не удалось — сообщение «Обратитесь к менеджеру за новым ключом».
3. Self-service запросов прав не будет (упрощение для хакатона).
4. В админ-панели: менеджмент пользователей, назначение прав доступа, статистика использования и заполненность хранилища.
5. Offline-доступ к `Private` файлам не поддерживается в MVP.

---

## 10) Короткое позиционирование (для презентации)

`Corporate Cloud` в MTC Cloud — это корпоративное файловое пространство поверх IaaS-инфраструктуры, где сотрудники безопасно подключаются по уникальной ссылке и ключу доступа, работают с личными и общими папками, а администратор управляет структурой и правами в отдельной панели.

## 11) Порядок реализации (рекомендуемый roadmap)

Принцип реализации:
- сначала полностью собрать UX-макет приложения на Flutter со stub-данными;
- затем подключить backend без ломки UI через слой репозиториев/сервисов.

### Фаза A — Flutter UI-first (макет + заглушки)

1. **Навигационный каркас**
  - Экран `AuthGate` → `Welcome` / `Login` / `Register`.
  - После входа: нижний nav `Файлы / + / Ещё`.
  - В `Ещё` условный пункт `Панель администратора` по флагу роли.

2. **Экран импорта по ссылке и ключу (stub)**
  - Поля `ссылка` + `ключ`, локальная валидация.
  - Stub-режим: успешный вход по тестовым данным, ошибки по заранее заданным кейсам.

3. **Экран `Файлы` (stub-данные)**
  - Секции: `Private` и корпоративные папки.
  - Список папок/файлов из локального mock JSON/моделей.

4. **Экран `+` (загрузка)**
  - Mobile: выбор файла через picker.
  - Windows popup: drag-and-drop в drop-зону.
  - Стейт-машина UI: `idle / uploading / success / error`.

5. **Экран `Ещё`**
  - Профиль, организация, выход, служебные пункты.
  - Вход в `Панель администратора` только для роли Manager/Admin.

6. **`Панель администратора` (stub)**
  - Вкладки/блоки: пользователи, права доступа, статистика, заполненность.
  - Все данные локальные (mock), без API.

7. **Единое состояние приложения (без backend)**
  - Ввести интерфейсы `AuthRepository`, `FilesRepository`, `AdminRepository`.
  - Реализация `Mock...Repository` для демо и теста UX.

### Фаза B — Техническая подготовка интеграции

8. **Контракт моделей и DTO**
  - Зафиксировать модели Flutter под будущие backend-ответы.
  - Добавить мапперы `DTO -> domain model`.

9. **Инфраструктура API-клиента**
  - `dio`, обработка ошибок, интерсептор токена.
  - Feature-flag: переключение `mock` / `real backend`.

### Фаза C — Backend MVP (после готовности API)

10. **Auth и организационная привязка**
   - Логин/регистрация.
   - Эндпоинт импорта по ссылке+ключу.
   - Ограничение: пользователь в одной организации.

11. **Файлы и права**
   - Список папок/файлов с разделением `Private` и корпоративных.
   - Назначение прав менеджером на папки.
   - Загрузка файлов (MinIO/S3-путь) + статусы.

12. **Админ API**
   - Менеджмент пользователей.
   - Выдача/изменение прав доступа.
   - Статистика использования и заполненность хранилища.

### Фаза D — Подключение реального backend в Flutter

13. **Замена mock-репозиториев на API-репозитории**
   - UI не меняется, меняется только источник данных.

14. **Сквозная проверка сценариев**
   - Регистрация/вход → импорт по ссылке/ключу → файлы → загрузка `+` → админка.
   - Проверка ролей: обычный пользователь не видит админ-раздел.

15. **Release-минимум для хакатона**
   - Стабильные happy-path сценарии.
   - Понятные ошибки (ключ/сеть/права).
   - Демо на iOS/Android + Windows popup.

## 12) Что нужно от backend

Стек: **REST API** (JSON), хранилище файлов — **MinIO** (S3-совместимое).

---

### 12.1 Авторизация и пользователи — **уже реализовано**

Полная JWT-авторизация в приложении подключена и работает. Ниже — зафиксированный контракт из swagger для справки.

**Схема токенов:**
- `access-token` — `Bearer JWT` в заголовке `Authorization`.
- `refresh-token` — `HttpOnly cookie` с именем `refresh_token`.

| Метод | Endpoint | Назначение |
|-------|----------|------------|
| `POST` | `/api/auth/register` | Регистрация. Возвращает access-token в заголовке, refresh-token в cookie |
| `POST` | `/api/auth/login` | Вход. Аналогично |
| `POST` | `/api/auth/refresh` | Обновить access-token (передаётся refresh cookie) |
| `POST` | `/api/auth/change_password` | Смена пароля. Аннулирует все сессии |
| `DELETE` | `/api/auth/logout` | Завершить текущую сессию |
| `DELETE` | `/api/auth/logout_all` | Завершить все сессии пользователя |
| `GET` | `/api/users` | Список пользователей |
| `GET` | `/api/users/{id}` | Профиль пользователя по id |
| `PATCH` | `/api/users/update_info` | Обновление профиля (email, имя, телефон) |
| `POST` | `/api/users` | Создать пользователя (admin, protected) |

**`CreateUserDto`** (register):
```json
{
  "username": "johndoe123",      // обязательный, min 3
  "password": "12345678",        // обязательный, min 8
  "email": "john@example.com",   // обязательный
  "firstName": "John",           // обязательный, min 2
  "lastName": "Doe",             // обязательный
  "surName": "Nikolayevich",     // nullable
  "phoneNumber": "+123456789"    // обязательный
}
```

**Модель `User`** (ответ):
```json
{
  "id": 1,
  "email": "john@example.com",
  "username": "johndoe123",
  "firstName": "John",
  "lastName": "Doe",
  "surName": "Nikolayevich",
  "phoneNumber": "+123456789",
  "created_at": "2026-03-01T10:44:26Z",
  "roles": [{ "id": 1, "role": "USER" }]
}
```

> **Примечание:** роль `USER` — стандартная. Для корпоративного администратора потребуется отдельная роль (см. 12.2).

---

### 12.2 Организация и импорт по ссылке — **требует реализации**

Нужно добавить роль `CORPORATE_ADMIN` в таблицу `roles` (сейчас есть только `USER`) и реализовать два endpoint'а.

| Метод | Endpoint | Назначение |
|-------|----------|------------|
| `POST` | `/api/org/join` | Подключить пользователя к организации по ссылке и ключу |
| `GET`  | `/api/org/me` | Получить информацию о текущей организации пользователя |

Тело `POST /api/org/join`:
```json
{
  "link": "https://cloud.example.com/invite/abc123",
  "accessKey": "XXXX-YYYY"
}
```

Ответ при успехе (`200`):
```json
{
  "organizationId": "org_001",
  "organizationName": "МТС Технологии",
  "role": "employee"
}
```

Ошибки (коды):
- `400` — невалидный формат ссылки или ключа.
- `403` — ключ неверный или истёк.
- `409` — пользователь уже привязан к другой организации.

### 12.3 Файлы и папки — **требует реализации**

| Метод | Endpoint | Назначение |
|-------|----------|------------|
| `GET`  | `/api/files/folders` | Список всех папок организации + `private` папка пользователя |
| `GET`  | `/api/files/folders/{folderId}` | Содержимое папки (файлы внутри) |
| `POST` | `/api/files/folders` | Создать новую папку (только для `CORPORATE_ADMIN`) |
| `DELETE` | `/api/files/folders/{folderId}` | Удалить папку (только для `CORPORATE_ADMIN`) |
| `GET`  | `/api/files` | Список последних/поиск файлов |
| `DELETE` | `/api/files/{fileId}` | Удалить файл |
| `GET`  | `/api/files/{fileId}/download` | Получить presigned URL для скачивания из MinIO |

Структура папки в ответе:
```json
{
  "id": "folder_001",
  "name": "Документы компании",
  "isPrivate": false,
  "ownerRole": "org",
  "createdAt": "2026-01-15T10:00:00Z"
}
```

Структура файла в ответе:
```json
{
  "id": "file_001",
  "folderId": "folder_001",
  "name": "onboarding.pdf",
  "sizeBytes": 204800,
  "mimeType": "application/pdf",
  "uploadedBy": "user_007",
  "updatedAt": "2026-03-01T14:22:00Z"
}
```

### 12.4 Загрузка файлов (через MinIO) — **требует реализации**

Рекомендуемый подход — **presigned upload URL**, чтобы клиент грузил напрямую в MinIO без проксирования через бекенд.

Шаг 1 — запросить presigned URL:

| Метод | Endpoint | Назначение |
|-------|----------|------------|
| `POST` | `/api/files/upload-url` | Получить presigned PUT URL для MinIO |

Тело запроса:
```json
{
  "folderId": "folder_001",
  "fileName": "report_q1.xlsx",
  "mimeType": "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet",
  "sizeBytes": 512000
}
```

Ответ:
```json
{
  "uploadUrl": "https://minio.example.com/bucket/org_001/folder_001/report_q1.xlsx?X-Amz-Signature=...",
  "fileId": "file_pending_xyz",
  "expiresInSeconds": 300
}
```

Шаг 2 — клиент делает `PUT` напрямую на `uploadUrl`.

Шаг 3 — подтвердить загрузку:

| Метод | Endpoint | Назначение |
|-------|----------|------------|
| `POST` | `/api/files/{fileId}/confirm` | Сообщить бекенду, что файл загружен в MinIO |

Ответ — полный объект файла (как в 12.3).

MinIO конфигурация (на стороне backend):
- Бакет: `corporate-cloud` (или per-org бакеты: `org-{orgId}`).
- Путь объекта: `{orgId}/{folderId}/{fileId}/{fileName}`.
- Lifecycle policy: опционально, для автоочистки незавершённых загрузок.

### 12.5 Права доступа на папки (ACL) — **требует реализации**

| Метод | Endpoint | Назначение |
|-------|----------|------------|
| `GET`  | `/api/admin/folders/{folderId}/acl` | Получить список прав на папку |
| `PUT`  | `/api/admin/folders/{folderId}/acl` | Установить права пользователей на папку |

Уровни доступа:
- `read` — только просмотр и скачивание.
- `write` — чтение + загрузка файлов.
- `manage` — полный доступ (включая удаление, переименование).

Тело `PUT`:
```json
{
  "permissions": [
    { "userId": "user_002", "level": "write" },
    { "userId": "user_003", "level": "read" }
  ]
}
```

### 12.6 Администраторский раздел — **требует реализации**

| Метод | Endpoint | Назначение |
|-------|----------|------------|
| `GET`  | `/api/admin/members` | Список участников организации с ролями |
| `PUT`  | `/api/admin/members/{userId}/role` | Изменить роль участника |
| `DELETE` | `/api/admin/members/{userId}` | Удалить участника из организации |
| `GET`  | `/api/admin/stats` | Статистика: файлы, папки, использование хранилища |

Ответ `/admin/stats`:

```json
  "storageTotalBytes": 536870912000,
  "fileCount": 47,
  "folderCount": 8,
  "memberCount": 12
}
```

### 12.7 Единая модель ошибок

Все ошибки возвращаются в виде:
```json
{
  "code": "INVALID_ACCESS_KEY",
  "message": "Ключ доступа неверный или истёк",
  "details": null
}
```

Коды ошибок, которые Flutter-клиент обязан обрабатывать:

| Код | HTTP | Описание |
|-----|------|----------|
| `INVALID_ACCESS_KEY` | 403 | Ключ доступа неверный или истёк |
| `ALREADY_IN_ORG` | 409 | Пользователь уже в другой организации |
| `FORBIDDEN` | 403 | Недостаточно прав для операции |
| `FILE_NOT_FOUND` | 404 | Файл или папка не найдены |
| `STORAGE_QUOTA_EXCEEDED` | 507 | Превышена квота хранилища |
| `UPLOAD_EXPIRED` | 410 | Presigned URL истёк, нужно запросить новый |
| `INVALID_TOKEN` | 401 | Токен недействителен, требуется повторный вход |

## 13) Definition of Done для этапа «UI со заглушками»

- Все ключевые экраны работают без backend.
- Ролевое поведение в UI видно и проверяется через mock-роль.

- Пользовательский поток от welcome до файлов и админки демонстрируется end-to-end.
